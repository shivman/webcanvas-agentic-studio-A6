The  logging  module

 

The  logging  module in Python is a built-in library used for tracking events that happen when your code runs. IT's mainly used for report status, errors, and debug information (no more  print ), making it easier to understand what's happening inside your application during development and after deployment. 

 

Why use  logging  instead or  print() ?

  • Better control over message level (info, warning, error, etc.)
  • Easy to route logs to files, streams, or external systems
  • Supports formatting and timestamps
  • Can be adjusted globally without changing every log statement.

Log levels
Python provides several log levels, each serving a different purpose:

  • DEBUG (10): Detailed information useful primarily for diagnosing issues.
    • DEBUG, INFO, WARNING, ERROR, CRITICAL
  • INFO (20): Confirmation that the application is functioning as expected.
    • INFO, WARNING, ERROR, CRITICAL
  • WARNING (30): A sign that something unexpected occurred or a potential problem might arise soon.
    • WARNING, ERROR, CRITICAL
  • ERROR (40): Indicates a significant issue that has prevented certain functions from executing.
    • ERROR, CRITICAL
  • CRITICAL (40): A severe error suggesting that the program may be unable to continue running.
    • CRITICAL only

The  pydantic  library: Agent's Personal Assistant

 

 pydantic  is a Python library used for data validation and settings management using Python type annotations. It allows you to define Python classes that act like schemas, ensuring that data passed around your system is structured, typed and validated. 

 

Agentic AI isn't just about "Call GPT and wait". It's modular cognition:

👁️ Perceive → 🧠 Remember → 🧭 Decide → 🎯 Act

 

Each step passed information to the next. And by now I am sure you know that this info is messy or malformed. That's where Pydantic steps in. It enforces clean, type-safe data across your cognitive layers.

 

In Agentic AI, especially when we're building modular cognitive layers (Perception, Memory, Decision, and Action), data flows between modules. You need a consistent, robust way to:

  • Define the shape of input/output between layers
  • Validate that agents are passing around expected structures
  • Serialize and trace agent memory over time
  • Ensure Clean interoperability between APIs, LLM Prompts and logs.

Single Agent vs multi-agent architectures

 

When building an agentic system you can build either a single-agent or a multi-agent architecture. 

 

The choice depends on your use case and how complex the required agent actions are:

 

Single Agent Architecture

 

Strengths:

  • Low complexity and easier to develop and manager
  • No coordination between multiple agents
  • May require fewer computational resources for a single powerful agent

Weakness:

  • May struggle with complex or dynamic tasks
  • Limited in handling tasks that require collaboration or diverse expertise
  • The agent can get confused and use an incorrect tool call arguments if the agent has too many different tool options available
  • May require a larger, more expensive model to handle multiple reasoning steps

 

Multi-Agent Architecture

Multi-agent architectures have multiple AI agents collaborating to resolve tasks. 

Note that each agent is equipped with its own memory here. However, you can also have memory for the compositional agentic architecture.

 

Strengths:

  • Capable of handling complex and dynamic tasks
  • Capable of parallel processing for efficiency
  • Possible to use smaller models specialized for distinct tasks

Weakness:

  • Increased complexity due to multiple agents collaborating with each other.
  • Requires robust mechanisms to manage interactions.
  • Harder to debug and optimize due to added complexity. 
  • May require more resources as more agents are added to the system. 

 

 

 

Patterns in multi-agent systems

 

Parallel
Multiple agents work simultaneously on different parts of a task.

 

 

Sequential
Tasks are processed sequentially, where one agent's output becomes input for the next.

 

 

 

Loop
Agents operate in iterative cycles, continuously improving their outputs based on feedback from other agents(s)

 

 

Router
A central router determines which agent(s) to invoke based on the task or input.

 

 

I really don't like these patterns, because I think the system should decide on its own what to do, or which pattern to pick.
